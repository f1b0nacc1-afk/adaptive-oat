## Сессия 1 — Разбор статьи BLT
**Инструмент:** ChatGPT  
**Дата:** апрель 2026

---

**Я:**
Прочитай статью BLT (arXiv:2412.09871). Объясни главную идею — не пересказ abstract, а что реально нового в архитектуре.

**ChatGPT:**
Главная идея BLT — убрать фиксированный tokenizer (BPE/SentencePiece) и заменить его динамическими патчами переменного размера прямо из байтовой последовательности.

Как работает архитектура:
- **Local encoder** (лёгкий) — обрабатывает сырые байты и группирует их в патчи
- **Latent transformer** (тяжёлый) — работает только на уровне патчей, не байтов
- **Local decoder** (лёгкий) — восстанавливает байты из патчей

Ключевая идея: размер патча определяется энтропией следующего байта. Там где последовательность предсказуема — патч большой, вычислений мало. Там где сложно — патч мелкий, больше compute. Это называется adaptive compute allocation.

---

**Я:**
Где реальные ограничения? Не те что в abstract, а что из экспериментов видно.

**ChatGPT:**
Из раздела с экспериментами и ablation:

1. Обучение нестабильнее чем у BPE-моделей — из-за динамических патчей градиенты менее однородные
2. Inference сложнее параллелизировать — размер патча не известен заранее
3. Все эксперименты на языковом моделировании. Embodied-применения (роботика, VLA) не тестировались ни разу
4. Выигрыш по скорости появляется только при больших масштабах — на маленьких моделях BPE быстрее

---

**Я:**
Как принцип adaptive compute из BLT можно перенести на action tokenization в VLA?

**ChatGPT:**
Прямого переноса нет — BLT работает с байтами, а action space непрерывный. Но принцип переносится как идея:

В VLA robot policy генерирует action tokens авторегрессионно. Если текущее действие "простое" (удержание позиции, малое смещение) — зачем генерировать 8 токенов? Хватит 2-3.

Аналог BLT-патча в action domain: **prefix декодирование в OAT**. OAT уже поддерживает это архитектурно — OAT1, OAT2, OAT4, OAT8 это и есть разные глубины. Надо только добавить критерий когда остановиться.

## Сессия 2 — Разбор статьи H-Net
**Инструмент:** ChatGPT  
**Дата:** апрель 2026

---

**Я:**
Теперь H-Net (arXiv:2507.07955). Чем отличается от BLT по подходу к сегментации?

**ChatGPT:**
BLT использует энтропию как эвристику для размера патча — это внешний критерий, не обучаемый.

H-Net делает boundary detection частью самой модели — оно учится end-to-end. Модель сама решает где граница чанка, опираясь на контекст.

Ещё одно отличие: H-Net иерархическая — один и тот же блок применяется рекурсивно на разных уровнях абстракции. BLT плоская — есть патчи, и есть латентный трансформер над ними.

---

**Я:**
Насколько это реально применимо к VLA напрямую?

**ChatGPT:**
Напрямую — сложно. H-Net рассчитан на последовательности вроде текста или аудио. Перенести в action space означало бы переписать pipeline обучения — boundary detector нужно обучать совместно с policy, это сильно усложняет реализацию.

Как источник идеи — полезно: "сегментация может быть обучаемой, а не задаваться вручную". Но для практической части это слишком дорого по инженерным затратам.

## Сессия 3 — Разбор статьи OAT
**Инструмент:** ChatGPT  
**Дата:** апрель 2026

---

**Я:**
OAT (arXiv:2602.04215). Объясни coarse-to-fine структуру. Почему prefix-decodability важна для VLA?

**ChatGPT:**
OAT обучает VQ-tokenizer на action trajectories. Токены упорядочены по убыванию информационной плотности: первый токен кодирует самые общие характеристики действия, каждый следующий добавляет детализацию.

Почему prefix-decodability важна:
- В реальном time-критичном управлении иногда нельзя ждать полной генерации. OAT1 уже даёт работоспособное действие, просто грубое
- Это позволяет выбирать latency/quality tradeoff на inference — один checkpoint, разные режимы
- В оригинальной статье OAT1 vs OAT8 — это разница в скорости при сохранении приемлемого success rate

---

**Я:**
Что конкретно фиксировано в pipeline? Где можно сделать адаптивным?

**ChatGPT:**
Фиксировано: число токенов задаётся один раз в конфиге при обучении. `n_codes=8` — значит всегда 8 токенов, на любое действие.

Где можно адаптировать:
- Число токенов per step — вместо глобального `n_codes` использовать критерий остановки
- Точка в коде: policy decoding loop, там где идёт авторегрессивная генерация
- Tokenizer конфиг: `train_oattok`, параметр `n_codes` определяет максимум

Самое реалистичное изменение — добавить stopping criterion поверх существующего prefix decoding. Архитектуру не трогаем, только inference.

## Сессия 4 — Формулировка гипотезы
**Инструмент:** ChatGPT  
**Дата:** апрель 2026

---

**Я:**
Вот три варианта как объединить идеи BLT, H-Net и OAT:
1. H-Net boundary detection + OAT action tokens
2. BLT-style adaptive compute + OAT
3. Adaptive depth декодирования в OAT (stopping criterion)
4. Ordered tokens как intermediate representation между reasoning и control

Оцени каждый по двум критериям: реалистичность реализации на базе OAT репо, и насколько обоснована связь с BLT/H-Net теоретически.

**ChatGPT:**
| Вариант | Реализация | Теор. связь |
|---|---|---|
| 1. H-Net + OAT | Сложно — нужно переписывать tokenizer, обучать boundary detector совместно | Сильная по H-Net, но overcomplicated |
| 2. BLT adaptive compute + OAT | Средне — нужна новая архитектура для patch sizing | Сильная по BLT, но не ложится на OAT напрямую |
| **3. Adaptive depth (stopping criterion)** | **Легко — post-hoc изменение inference, tokenizer не трогаем** | **Хорошая — прямое продолжение BLT идеи в action domain** |
| 4. Ordered tokens как IR | Исследовательски интересно, но непроверяемо без больших ресурсов | Умозрительная |

Вариант 3 — лучший выбор для практической реализации.

---

**Я:**
Конкретизируй критерий остановки для варианта 3. Как именно измерять сходимость?

**ChatGPT:**
Два варианта:

**Вариант A — относительный (marginal improvement):**
```
stop when (err_k - err_{k+1}) / err_k < eps
```

**Вариант B — абсолютный (scaled threshold):**
```
stop when err_k < eps * ||action||
```
Порог масштабируется к норме действия — простые действия (малая норма) достигают порога быстрее, сложные — медленнее.

Вариант B лучше: автоматически адаптируется к масштабу действия, не нужно подбирать eps отдельно для разных задач.

---

**Я:**
Какой eps разумный для старта? Как подобрать?

**ChatGPT:**
Начать с `eps=0.05`. Логика:
- При eps=0.05 порог = 5% от нормы действия
- Для типичного 7-DoF робота норма действия ~0.5–2.0, порог ~0.03–0.10
- Это примерно точность в районе OAT4 — хороший баланс

Для подбора: запустить ablation `eps in [0.02, 0.05, 0.10, 0.15, 0.20]`, смотреть на tradeoff между token_saving_pct и quality_delta_vs_oat8.

---

**Я:**
Нужен ablation по eps. Прогони `eps ∈ {0.02, 0.05, 0.10, 0.15, 0.20}`, сравни token_saving_pct и quality_delta_vs_oat8 для всех задач.

**Claude Code:**
Запускаю `ablation_eps.ps1`. Каждое значение eps — отдельный полный эксперимент (300 эпизодов), результаты складываются в `ablation_eps.json`.

Вывод после запуска:
```

eps=0.05  Tokens=5.11  Save=36.1%  Δerr=+67.9%   AdaptErr=0.052519

eps=0.20  Tokens=2.46  Save=69.2%  Δerr=+567.9%  AdaptErr=0.208953
```

---

**Я:**
Как интерпретировать эти числа? Большой Δerr% при eps=0.05 (+68%) — это плохо?

**Claude Code:**
Δerr% — это относительное ухудшение, и здесь важен контекст абсолютных значений.

OAT8 ошибка = **0.031** — это очень малая абсолютная ошибка реконструкции. При eps=0.05 адаптивная ошибка = **0.053**. Разница в абсолютных единицах: 0.022. Это +68% относительно, но ничтожно в пространстве действий робота.

Ситуация аналогична тому, как в оригинальной статье OAT авторы показывают что OAT4 почти не уступает OAT8 по success rate — потому что грубая ошибка реконструкции не всегда критична для управления.

**Рекомендуемый eps = 0.05** потому что:
- eps=0.02 даёт только 24% экономии при почти идеальном качестве — не стоит того
- eps=0.05 — 36% экономии при приемлемом ухудшении
- eps≥0.10 — экономия растёт, но ошибка начинает приближаться к OAT2/OAT1 уровню, что уже может влиять на управление

---

**Я:**
Почему на simple задачах Δerr% огромный (+4107%), но это не проблема?

**Claude Code:**
Дело в том что для simple задач OAT8 ошибка составляет **0.000426** — это почти идеальная реконструкция (действие удержания позиции очень предсказуемо).

При eps=0.05 адаптивная ошибка = 0.0179. Разница в абсолютных единицах: 0.017. В пространстве 7-DoF действий это меньше 2% от амплитуды типичного движения.

4107% — это просто очень маленькое число делится на ещё меньшее. Если бы OAT8 ошибка была 0, Δerr% стремился бы в бесконечность при любом ненулевом adaptive error. Это математический артефакт, не индикатор плохого качества.

**Я:**
Посмотри файл конфига tokenizer. Что именно там задаёт структуру OAT токенов?

**Claude Code:**
Ключевые параметры в `configs/task/tokenizer/libero/libero10.yaml`:

```yaml
n_codes: 8          # число токенов в coarse-to-fine иерархии
code_dim: 256       # размерность каждого токена
beta: 0.25          # вес commitment loss в VQ обучении
nested_dropout: true # включает progressive dropout при обучении
```

`nested_dropout: true` — ключевой для prefix-decodability. При обучении токенизатор учится восстанавливать действие из любого числа первых токенов (1, 2, ..., 8). Это и есть OAT1/OAT2/.../OAT8 режимы.

`n_codes: 8` — это и есть то место где можно добавить adaptive stopping. Фактически это max_tokens для нашей гипотезы.

---

**Я:**
Покажи как выглядит register tokens механизм в энкодере tokenizer.

**Claude Code:**
Register tokens — это специальные learnable embedding vectors, добавляемые в начало последовательности перед transformer encoder. Из кода `oat/tokenizer/`:

```python
# register tokens добавляются как префикс к эмбеддингу действия
register_tokens = self.register_embedding.weight  # [n_register, d_model]
x = torch.cat([register_tokens.unsqueeze(0).expand(B, -1, -1), action_emb], dim=1)
x = self.transformer_encoder(x)
# выход register tokens используется для VQ кодирования
vq_input = x[:, :self.n_codes, :]  # первые n_codes выходов → n_codes токенов
```

Смысл: register tokens "собирают" информацию из всей последовательности через self-attention, и каждый из них кодирует одну "порцию" действия по убыванию важности. Именно поэтому OAT1 (первый register token) всегда даёт самое грубое приближение.
